# 股票分析系统 — 对外 Skill 服务设计文档

> 版本：v1.0
> 日期：2026-03-21
> 定位：将多智能体股票分析系统封装为标准 HTTP Skill 服务，供其他 AI 智能体调用

---

## 一、系统定位与边界

### 1.1 这个系统是什么

本系统是一个**以 Skill 形式对外暴露的多智能体股票分析引擎**。外部 AI 智能体（例如 Claude Code）通过调用预定义的 HTTP 接口，提交股票分析任务，并异步获取最终生成的 Markdown 分析报告。

系统内部维护完整的四阶段分析流水线，由编排层自动驱动，外部调用方无需感知内部运作细节。

### 1.2 外部 AI 的使用模式

```
外部 AI 智能体（调用方）
        │
        │  ① 读取 Skill 文件，了解接口规范
        ↓
  Stock Analysis Skill
  ┌──────────────────────────────┐
  │  POST /api/v1/tasks          │  ← 创建分析任务，获得 task_id
  │  GET  /api/v1/tasks/{id}     │  ← 轮询进度，查看当前阶段
  │  GET  /api/v1/tasks/{id}/report │  ← 获取最终 MD 报告
  │  DELETE /api/v1/tasks/{id}   │  ← 取消任务
  │  GET  /api/v1/tasks          │  ← 列出所有任务
  └──────────────────────────────┘
        │
        ↓  内部自动执行
  四阶段多智能体流水线
  Stage 1 → Stage 2 → Stage 3 → Stage 4
        │
        ↓  输出
  Markdown 分析报告（保存至 reports/ 目录）
```

### 1.3 核心约束

| 约束项 | 说明 |
|-------|------|
| 无数据库 | 任务状态以 JSON 文件存储，无需安装任何数据库 |
| 无 Redis | 任务队列基于进程内内存队列，无需 Redis |
| 无前端 | 纯后端服务，通过 HTTP API 与外部 AI 交互 |
| 单用户 | 面向单一用户/单一部署实例，无多租户设计 |
| 一键安装 | 依赖 uv 包管理器，一条命令完成所有依赖安装 |

### 2.2 外部 Skill 文件与本系统的关系

| 名称 | 说明 |
|-----|------|
| **外层 Skill 文件** | 供外部 AI（如 Claude Code）读取的 Markdown 接口说明文件，描述如何调用本系统的 HTTP API。这是本系统对外暴露的"能力说明书"。 |

## 三、对外 Skill 文件设计

### 3.1 Skill 文件的作用

外部 AI 智能体（调用方）在使用本系统前，先读取 Skill 文件（`stock-analysis-skill.md`）。该文件告知外部 AI：

- 本系统能做什么
- 何时应该调用本系统
- 如何构造 HTTP 请求
- 如何轮询任务进度
- 如何处理各种状态
- 如何读取最终报告

### 3.2 Skill 文件内容结构

```
stock-analysis-skill.md
│
├── 技能名称与用途说明
│     描述：分析哪支股票时应调用本技能
│
├── 调用前置条件
│     服务地址确认（默认 http://127.0.0.1:8080）
│     支持的市场类型（A股 / 港股 / 美股）
│     股票代码格式规范（000001.SZ / 00700.HK / AAPL）
│
├── 接口调用流程（步骤说明）
│     Step 1：创建任务 → 获取 task_id
│     Step 2：轮询进度（建议间隔 30 秒）
│     Step 3：状态为 completed → 获取报告
│     Step 4：出现错误 → 读取错误信息，决定是否重试
│
├── 各接口详细说明
│     POST /api/v1/tasks
│     GET  /api/v1/tasks/{task_id}
│     GET  /api/v1/tasks/{task_id}/report
│     DELETE /api/v1/tasks/{task_id}
│     GET  /api/v1/tasks
│     GET  /api/v1/health
│
├── 任务状态说明
│     pending / running / completed / failed / cancelled
│
├── 轮询建议策略
│     阶段进度参考完成时间（供外部 AI 决定轮询频率）
│
└── 报告读取说明
      报告为 Markdown 格式，包含操作建议 / 价格区间 / 止损位 / 操作步骤
```

### 3.3 外部 AI 调用完整流程图

```
外部 AI 智能体
      │
      │  用户请求："帮我分析一下 000001.SZ"
      ↓
  读取 stock-analysis-skill.md（了解接口规范）
      │
      ↓
  POST /api/v1/tasks
  请求体：{ "stock_code": "000001.SZ" }
      │
      ↓  返回 task_id
  task_id = "TASK_20260321_143022_ABC123"
      │
      ↓  等待 30 秒
  GET /api/v1/tasks/TASK_20260321_143022_ABC123
      │
      ├── status = "running" → 继续等待，再次轮询
      │
      └── status = "completed"
              │
              ↓
          GET /api/v1/tasks/TASK_20260321_001/report
              │
              ↓  返回完整 Markdown 报告
          将报告内容呈现给用户
```

## 四、HTTP API 设计

### 4.1 接口总览

| 方法 | 路径 | 功能 |
|-----|------|------|
| POST | `/api/v1/tasks` | 创建分析任务 |
| GET | `/api/v1/tasks` | 查询任务列表 |
| GET | `/api/v1/tasks/{task_id}` | 查询单个任务详情与进度 |
| DELETE | `/api/v1/tasks/{task_id}` | 取消任务 |
| GET | `/api/v1/tasks/{task_id}/report` | 获取任务最终报告 |
| GET | `/api/v1/health` | 服务健康状态 |

### 4.2 创建任务

**请求：POST /api/v1/tasks**

| 请求字段 | 类型 | 必填 | 说明 |
|---------|------|-----|------|
| stock_code | 字符串 | 是 | 股票代码，如 `000001.SZ`、`00700.HK`、`AAPL` |
| note | 字符串 | 否 | 可选备注，用于标识该次分析目的 |

**响应字段：**

| 响应字段 | 类型 | 说明 |
|---------|------|------|
| task_id | 字符串 | 任务唯一标识 |
| status | 字符串 | 初始状态为 `pending` |
| stock_code | 字符串 | 回显股票代码 |
| queue_position | 整数 | 当前在队列中的位置（1 表示立即执行）|
| created_at | 字符串 | 任务创建时间（ISO 8601）|

### 4.3 查询任务进度

**请求：GET /api/v1/tasks/{task_id}**

**响应字段：**

| 响应字段 | 类型 | 说明 |
|---------|------|------|
| task_id | 字符串 | 任务 ID |
| status | 字符串 | 当前状态（见任务状态说明）|
| stock_code | 字符串 | 分析标的 |
| current_stage | 字符串 | 当前执行阶段（stage1 / stage2 / stage3 / stage4）|
| stage_progress | 对象 | 各阶段状态（pending / running / completed / skipped）|
| created_at | 字符串 | 创建时间 |
| started_at | 字符串 | 开始执行时间（未开始为 null）|
| completed_at | 字符串 | 完成时间（未完成为 null）|
| error | 字符串 | 错误信息（无错误为 null）|
| report_path | 字符串 | 报告文件路径（未完成为 null）|

**stage_progress 示例结构：**

```
{
  "stage1": "completed",       ← 六维分析（并行）
  "stage2": "running",         ← 多空研判与交易计划（混合）
  "stage3": "pending",         ← 量化风控（混合）
  "stage4": "pending"          ← 投资顾问报告（串行）
}
```

### 4.4 获取报告

**请求：GET /api/v1/tasks/{task_id}/report**

- 仅当 status 为 `completed` 时可成功获取
- 其他状态返回 HTTP 400，携带提示信息

**响应字段：**

| 响应字段 | 类型 | 说明 |
|---------|------|------|
| task_id | 字符串 | 任务 ID |
| stock_code | 字符串 | 分析标的 |
| content | 字符串 | 完整 Markdown 报告内容 |
| report_path | 字符串 | 报告在服务器上的文件路径 |
| completed_at | 字符串 | 分析完成时间 |

### 4.5 取消任务

**请求：DELETE /api/v1/tasks/{task_id}**

- 若 task_id 不存在：返回 HTTP 404
- 若任务处于 `pending` 状态：立即从队列移除，状态更新为 `cancelled`
- 若任务处于 `running` 状态：发送取消信号，当前正在调用的 LLM 请求完成后停止，不再进入下一阶段，状态更新为 `cancelled`
- 若任务已 `completed` / `failed`：返回 HTTP 400，提示无法取消
- 取消后任务 JSON 文件保留（保留用于排查问题），已生成的中间数据不清理

### 4.6 查询任务列表

**请求：GET /api/v1/tasks**

| 查询参数 | 类型 | 说明 |
|---------|------|------|
| status | 字符串 | 按状态过滤，可选：pending / running / completed / failed / cancelled |
| limit | 整数 | 返回条数，默认 20，最大 100 |
| offset | 整数 | 分页偏移，默认 0 |

### 4.7 健康检查

**请求：GET /api/v1/health**

**响应字段：**

| 响应字段 | 类型 | 说明 |
|---------|------|------|
| status | 字符串 | `ok` 或 `degraded` |
| queue_size | 整数 | 当前队列中等待的任务数 |
| running_tasks | 整数 | 当前正在执行的任务数 |
| max_concurrent_tasks | 整数 | 当前配置允许的最大并发任务数（由 ENV 计算得出）|

## 五、任务生命周期

### 5.1 任务状态机

```
                   ┌─────────────────────────────────────┐
                   │           任务状态机                  │
                   └─────────────────────────────────────┘

         创建任务
            │
            ▼
        ┌─────────┐
        │ PENDING │  ← 在队列中等待，未开始执行
        └─────────┘
            │
            │  Worker 取走任务
            ▼
        ┌─────────┐
        │ RUNNING │  ← 流水线正在执行中
        └─────────┘
            │
     ┌──────┴──────┬──────────────┐
     │             │              │
     ▼             ▼              ▼
┌──────────┐  ┌────────┐  ┌──────────────┐
│COMPLETED │  │ FAILED │  │  CANCELLED   │
│ 分析完成  │  │ 出错   │  │  已取消      │
└──────────┘  └────────┘  └──────────────┘

注：CANCELLED 可从 PENDING 或 RUNNING 触发（DELETE 接口）
    FAILED 仅从 RUNNING 触发（流水线内部异常）
```

### 5.2 任务状态转换触发条件

| 从 | 到 | 触发条件 |
|---|---|---------|
| — | PENDING | 调用 POST /api/v1/tasks 成功 |
| PENDING | RUNNING | Worker 从队列取出任务并开始执行 |
| PENDING | CANCELLED | 外部调用 DELETE 接口 |
| RUNNING | COMPLETED | Stage 4 投资顾问输出报告，写入文件成功 |
| RUNNING | FAILED | 任意阶段发生不可恢复的错误（LLM 调用失败超出重试次数、数据源异常等）|
| RUNNING | CANCELLED | 外部调用 DELETE 接口，当前子任务完成后停止 |

### 5.3 停牌特殊处理

Stage 3 进入前，编排层先执行停牌前置检查：

```
Stage 2 完成
      │
      ▼
  停牌检查（纯代码）
      │
      ├── 当日停牌？
      │     是 → 直接生成"禁止交易"提示报告
      │          Stage 3 全部 AI 智能体不启动
      │          task_id 状态标记为 completed（报告已生成）
      │
      └── 未停牌 → 继续进入 Stage 3 正常流程
```

## 六、并发模型

### 6.1 三层并发结构

本系统存在三个独立的并发维度，互不冲突但相互制约：

```
┌─────────────────────────────────────────────────────────────────┐
│                       三层并发结构                               │
│                                                                 │
│  第一层：模型 API 全局并发上限                                    │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  全局信号量：同一时刻最多允许 N 个 LLM API 调用在飞        │    │
│  │  配置项：MODEL_API_MAX_CONCURRENCY（例：10）              │    │
│  └─────────────────────────────────────────────────────────┘    │
│                           ↕ 受制于                              │
│  第二层：单任务内部并发上限                                       │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  每个任务内：并行智能体最多同时占用 M 个 API 调用名额      │    │
│  │  配置项：TASK_MAX_MODEL_CONCURRENCY（例：2）              │    │
│  └─────────────────────────────────────────────────────────┘    │
│                           ↕ 推导出                              │
│  第三层：最大并发任务数（自动计算，无需手动设置）                  │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  MAX_CONCURRENT_TASKS = MODEL_API_MAX_CONCURRENCY        │    │
│  │                       ÷ TASK_MAX_MODEL_CONCURRENCY       │    │
│  │  示例：10 ÷ 2 = 5（最多同时运行 5 个分析任务）            │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 任务队列机制

```
外部 AI 不断提交任务（可能 100 个）
           │
           ▼
    ┌─────────────┐
    │  任务队列    │  ← asyncio.Queue，上限由 TASK_QUEUE_MAX_SIZE 控制
    │  FIFO 排队  │    超出上限时，新提交任务返回 HTTP 429（队列已满）
    └─────────────┘
           │
           │  Worker 池持续消费队列
           ▼
    ┌─────────────────────────────────┐
    │  Worker 池（并发执行任务）         │
    │  同时最多运行 MAX_CONCURRENT_TASKS│
    │  个任务（由并发信号量控制）         │
    └─────────────────────────────────┘
           │
    ┌──────┴──────┬──────┴──────┐
    ▼             ▼             ▼
  任务 A        任务 B        任务 C
（执行中）     （执行中）     （执行中）
```

### 6.3 并发示例说明

配置：MODEL_API_MAX_CONCURRENCY = 10，TASK_MAX_MODEL_CONCURRENCY = 2

- 最多同时运行 5 个任务
- 每个任务的 Stage 1（6 个并行分析师）：最多同时发出 2 个 LLM API 调用，分 3 批完成
- 5 个任务同时在 Stage 1 时：共占用 5 × 2 = 10 个 API 调用名额，恰好打满全局上限
- 任意时刻，全局 LLM API 调用总数不会超过 10

## 七、四阶段流水线编排

### 7.1 完整流水线流程图

```
任务开始
   │
   ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 1：六维并行分析                                     │
│                                                          │
│  数据注入  →  并行启动 6 个分析师                           │
│              ┌──────┐ ┌──────┐ ┌──────┐                  │
│              │技术  │ │基本面│ │微观  │                    │
│              │分析师│ │分析师│ │结构  │  → 各输出           │
│              └──────┘ └──────┘ └──────┘    0-100 评分     │
│              ┌──────┐ ┌──────┐ ┌──────┐    + Markdown     │
│              │市场  │ │板块  │ │资讯  │    分析报告         │
│              │情绪  │ │轮动  │ │事件  │                    │
│              └──────┘ └──────┘ └──────┘                  │
│  受 TASK_MAX_MODEL_CONCURRENCY 控制，分批并行执行           │
└──────────────────────────────────────────────────────────┘
   │  全部 6 份报告收集完成
   ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 2：多空研判与交易计划（混合）                         │
│                                                          │
│  第一步：并行阶段                                          │
│    看涨分析师  ←→  看跌分析师（同时启动，各自基于 Stage 1）   │
│    辩论 1-2 轮（看涨方读取看跌论点后反驳，反之亦然）           │
│                                                          │
│  第二步：串行阶段                                          │
│    研究主管：综合六维评分 + 辩论记录 → 给出方向与置信度       │
│        ↓                                                 │
│    交易计划师：将裁决转化为具体交易参数（入场/止损/止盈/仓位） │
└──────────────────────────────────────────────────────────┘
   │  交易计划书产出完成
   ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 3：量化风控（混合）                                  │
│                                                          │
│  Step 1（纯代码，编排层执行）：                             │
│    停牌前置检查 → 若停牌，直接输出禁止交易报告，跳过全部 AI   │
│    VaR 计算（历史模拟法）                                   │
│    A股特有风险评分（T+1 / 涨跌停 / 停牌 / 融资融券）         │
│    综合风险分级（<50 / 50-69 / ≥70）                       │
│                ↓  计算结果注入                             │
│  Step 2（AI 并行）：                                       │
│    激进风控师  ←→  保守风控师  ←→  量化风控师（同时启动）     │
│                ↓  三方报告汇总                             │
│  Step 3（AI 串行）：                                       │
│    首席风控官：综合三方意见，拥有一票否决权，给出最终裁决       │
└──────────────────────────────────────────────────────────┘
   │  首席风控官裁决完成
   ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 4：投资顾问报告                                     │
│                                                          │
│  输入：CRO 裁决 + 交易计划书 + 六维评分汇总 + 价格快照       │
│  输出：面向普通投资者的 Markdown 报告                       │
│    - 单一操作建议（买入/卖出/持仓/观望）                     │
│    - 明日价格参考区间                                       │
│    - 7日价格走势参考区间                                    │
│    - 关键数字（止损价 / 目标价 / 建议仓位）                  │
│    - 3 条核心理由（无专业术语）                              │
│    - 2 条主要风险提示                                       │
│    - 分步操作指南                                           │
└──────────────────────────────────────────────────────────┘
   │  报告写入 reports/ 文件
   ▼
任务状态 → COMPLETED
```

### 7.2 阶段间数据传递原则

- 编排层负责收集每个阶段的输出，并作为下一阶段的输入注入
- 智能体只处理自己阶段的输入/输出，不直接访问其他阶段的数据
- 数据传递路径：各阶段输出 → 编排层缓存（内存）→ 下一阶段注入

### 7.3 数据工具注入时机

```
编排层启动某智能体之前：
  1. 读取该智能体的数据工具配置（需要哪些数据）
  2. 检查数据源实际可用性（Tushare 积分 / AkShare 可达性）
  3. 拉取可用数据，对不可用数据注入标准占位符说明
  4. 将数据注入智能体上下文，再启动 LLM 调用
```

## 九、环境配置（ENV 文件）

所有配置项通过项目根目录的 `.env` 文件设置。首次安装时，复制 `.env.example` 并填写实际值。

### 9.1 LLM API 配置

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| LLM_API_BASE | `https://api.openai.com/v1` | LLM API 地址，兼容所有 OpenAI 格式接口（含本地 Ollama、vLLM 等）|
| LLM_API_KEY | 无 | API 密钥（本地模型可填任意字符串）|
| LLM_MODEL_ID | `gpt-4o` | 模型 ID，按所用服务商填写 |
| LLM_TEMPERATURE | `0.3` | 生成温度，建议保持低温以提升分析稳定性 |
| LLM_MAX_TOKENS | `4096` | 单次 LLM 响应最大 token 数 |
| LLM_TIMEOUT_SECONDS | `120` | 单次 LLM 调用超时时间（秒）|
| LLM_MAX_RETRIES | `3` | LLM 调用失败后的最大重试次数 |

### 9.2 数据源配置

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| TUSHARE_TOKEN | 无 | Tushare Pro API Token |
| AKSHARE_ENABLED | `true` | 是否启用 AkShare 作为备用/补充数据源 |

### 9.3 并发配置

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| MODEL_API_MAX_CONCURRENCY | `10` | 全局 LLM API 最大并发调用数（按所用服务商的 RPM 限制设置）|
| TASK_MAX_MODEL_CONCURRENCY | `2` | 单个任务内部最大 LLM 并发数（控制每个任务对 API 的占用量）|
| TASK_QUEUE_MAX_SIZE | `100` | 任务队列最大容量（超出后新请求返回 429）|

> **说明**：`MAX_CONCURRENT_TASKS` 计算时采用**向下取整**（floor）。例如：MODEL_API_MAX_CONCURRENCY = 10，TASK_MAX_MODEL_CONCURRENCY = 3，则 MAX_CONCURRENT_TASKS = floor(10 ÷ 3) = 3，剩余 1 个 API 调用名额作为缓冲，用于各任务之间的轻量调用（如数据工具请求）。

> **说明**：`MAX_CONCURRENT_TASKS`（最大并发任务数）由系统自动计算，无需手动设置，计算方式为 `floor(MODEL_API_MAX_CONCURRENCY ÷ TASK_MAX_MODEL_CONCURRENCY)`，结果向下取整。

### 9.4 服务配置

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| SERVER_HOST | `127.0.0.1` | 服务监听地址（仅本机访问）|
| SERVER_PORT | `8080` | 服务监听端口 |

### 9.5 存储配置

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| TASKS_DIR | `./tasks` | 任务状态 JSON 文件存放目录 |
| REPORTS_DIR | `./reports` | 分析报告 Markdown 文件存放目录 |
| TASK_FILE_RETENTION_DAYS | `30` | 任务状态文件保留天数（超过自动清理）|

## 十、技术栈

### 10.1 最终选型总览

| 层级 | 选型 | 版本 | 选型理由 |
|-----|------|-----|---------|
| 编程语言 | Python | 3.12 | 数据源 SDK、AI 生态均以 Python 为主 |
| 包管理 | uv | 最新稳定版 | 比 pip 快 10 倍，一键管理依赖，支持虚拟环境 |
| HTTP 框架 | FastAPI | 最新稳定版 | 原生异步、性能好、自动生成 API 文档、教程最多 |
| LLM 调用 | openai SDK | 最新稳定版 | 行业标准 OpenAI 兼容接口，支持全部主流 LLM 服务商和本地模型 |
| 异步框架 | asyncio（Python 标准库）| 内置 | 零额外依赖，成熟稳定，全网教程极多 |
| 任务队列 | asyncio.Queue（标准库）| 内置 | 零额外依赖，单进程内高效，满足 100+ 任务排队需求 |
| 配置管理 | python-dotenv | 最新稳定版 | 轻量，读取 .env 文件的事实标准库 |
| 数据源 | Tushare Pro + AkShare | 最新稳定版 | 完整覆盖 A 股 / 港股 / 美股数据 |
| 流程编排 | 纯 Python asyncio 代码 | — | 流程固定，无需动态编排框架，代码即流程，直观可读 |
| 任务存储 | JSON 文件 | — | 零依赖，人类可读，单用户场景性能充裕 |
| 报告存储 | Markdown 文件 | — | 天然适配 AI 读写，便于外部 AI 解析 |

### 10.2 为何不用 LangChain / LangGraph

| 方案 | 结论 | 理由 |
|-----|------|------|
| LangGraph | 不采用 | 将固定流程抽象为"有向图"概念，增加不必要的复杂度；适合动态、可修改的流程，与本系统固定四阶段流水线不匹配 |
| LangChain | 不采用 | 其核心价值（LCEL 链式调用、各类 Runnable 抽象）在本系统中没有增益；版本变化频繁，历史教程兼容性差；直接用 openai SDK + 纯 Python 更清晰 |
| openai SDK + asyncio | 采用 | 每个智能体就是一次 LLM API 调用，Stage 1 并行就是 `asyncio.gather()`，Stage 2 串行就是顺序 `await`，流程透明，任何 Python 教程均适用 |

## 十二、关键设计决策说明

### 12.1 选择 HTTP 而非 CLI 的理由

- HTTP 接口对调用方（外部 AI）无环境要求，任何具备 HTTP 调用能力的 AI 框架均可使用
- HTTP 支持真正的异步任务模式：提交后立即返回 task_id，后续轮询进度，符合长时间分析任务的调用场景
- HTTP 接口规范（路径、请求体、响应结构）可完整写入 Skill 文件，外部 AI 无需安装任何本地依赖

### 12.2 选择文件存储而非数据库的理由

- 单用户场景下，并发写入量极低，JSON 文件性能完全充裕
- 零安装依赖，降低部署门槛
- 任务状态文件人类可读，便于排查问题
- 报告为 Markdown 文件，天然适合 AI 读取

### 12.3 选择 asyncio.Queue 而非 Celery/Redis 的理由

- 单进程内队列完全满足需求（单用户，队列深度 ≤ 100）
- 零外部依赖，无需安装和运维 Redis
- 任务取消（PENDING 阶段）可以直接在队列层面处理，无需跨进程通信

### 12.4 流程编排不使用框架的理由

- 四阶段流水线是完全固定的，不需要动态改变
- 纯 Python async/await 写出的流程比任何框架的 DSL 更直观
- 减少学习成本，便于后续维护和调试
